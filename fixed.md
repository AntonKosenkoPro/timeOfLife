9. extractIP mangled IPv6 RemoteAddr and trusted forwarded headers unconditionally.
   - Two related defects in `backend/internal/handlers/auth.go`. (a) `extractIP` split `RemoteAddr` on the last colon, truncating IPv6 literals like `[::1]:1234` and producing a bogus rate-limit key. Now uses `net.SplitHostPort`. (b) `extractIP` and chi's `RealIP` middleware honoured `X-Forwarded-For`/`X-Real-IP` unconditionally, so a client not behind a trusted proxy could spoof the header and get a fresh rate-limit key on every request â€” bypassing the OTP-request, OTP-verify, and Apple limiters. `RealIP` is removed; new `Handler.clientIP` honours forwarded headers only when the direct TCP peer is in the configured `TRUSTED_PROXIES` CIDR allowlist (env var, empty = trust nobody, the safe default). `extractIP` is now the raw peer helper. Covered by `TestExtractIP_*`, `TestClientIP_TrustedProxyGate`, `TestParseTrustedProxies`, and `TestRequestOTP_SpoofedHeaderDoesNotBypassRateLimit`. (auth-audit P0-1/P0-2; FURPS R1/S5/+2)

10. Concurrent 401 refreshes could mass-revoke all of a user's sessions.
    - The server rotates (and revokes) the refresh token on every refresh. Two overlapping 401s (e.g. `/me` from `restoreSession` plus a parallel authed request, or several requests 401ing at once) each triggered a separate refresh; the second used the just-revoked token, tripped the server's reuse-detection, and revoked every session for the user â€” a benign race that logged the user out on all devices. `AuthService.performRefresh` now coalesces concurrent callers onto one in-flight `Task` (stored before the first `await`, so a mid-refresh caller joins instead of starting its own); `restoreSession`'s manual 401-retry routes through `performRefresh` instead of calling `repository.refresh` directly. Covered by the new `performRefreshSingleFlight` test (holds the refresh on a gate, asserts two concurrent calls produce exactly one network refresh); all 138 iOS tests green. (auth-audit P0-3; FURPS R1/U3)
